¶关于 V2Ray（V2Fly）、XRay、VMess 、VLESS、XTLS 的说明

V2Ray（V2Fly）和 XRay 项目：

  • V2Ray（V2Fly）的核心工具是 V2Ray-Core，生态项目社区是 Project V 。

    V2Ray 作者失联于是大家组织成立了 V2Fly 项目，接管了 V2Ray 的后续更新。也就是说 V2Fly 就是现在的 V2Ray

  • XRay 的核心工具是 XRay-Core，功能上是 V2Ray-Core 的超集，生态项目社区是 Project X 。

    由于 XTLS 被 V2Fly 从 V2Ray-Core 中移除后，XTLS 协议的作者独立成立了 XRay 项目

加密传输协议：

  • VMess 是 V2Ray（V2Fly）旧版加密传输协议，VLESS 是 V2Ray（V2Fly）和 XRay 新版加密传输协议（V2Ray 和 XRay 两者都支持 VMess 、VLESS）。

  • XTLS 是一种应对两层 TLS 加密进行性能优化的一种协议（解决 TLS in TLS 问题）。注意：XTLS 目前只有 XRay 支持。

    原理是：将代理过程中的两层 TLS 合并成一层 TLS，减少一次 TLS 加密过程来达到提升性能，具体实现参考 XTLS 的 Vision 流控（xtls-rprx-vision）。

参考：https://www.chengxiaobai.com/essays/v2ray-trojan-xray.html

¶VMESS + WebSocket + TLS

由于 V2ray 原始的 VMESS 协议格式是固定的，GFW 可能会通过深度包检测（DPI）等技术分析流量的协议特征来判断流量是否属于代理，进而对 IP 或者端口进行封锁或限制。为了防止 VMESS 协议被识别，可以将代理流量混淆为其他常用的协议（如：WebSocket（HTTP）、WebSocket + TLS（HTTPS ）等）从而避免被探测识别。

  • VMESS + TCP + TLS：混淆 VMESS 流量 为 TLS 加密的 TCP 流量（带伪装的 V2ray，不能过 CDN 中转）
  • VMESS + WebSocket + TLS：混淆 VMESS 流量 为 TLS 加密的 WebSocket 流量（即最通用的 V2ray 伪装方式，能过 CDN 中转，推荐使用）

¶解决 V2ray 中 TIT 特征问题

¶TIT 特征问题简要说明

由于 VMESS + TCP + TLS、VMESS + WebSocket + TLS、VLESS + TCP + TLS、VLESS + WebSocket + TLS 等混淆方式都存在 TIT 特征问题，该特征已经能够被 GFW 精准识别，为了解决该问题，可以使用以下方式进行解决：

  • （1）使用 CDN 中转 V2ray 的 WebSocket 代理流量（可解决 VMESS + WebSocket + TLS 、VLESS + WebSocket + TLS 的 TIT 特征问题）
  • （2）使用 XTLS 的 Vision 流控（可解决 VLESS + TCP + TLS 的 TIT 特征问题）

3）XTLS-Vision 原理

原理参考：https://github.com/XTLS/Xray-core/discussions/1295

关于原理的补充说明：Vision 的代理客户端仅在第一个内层加密包中插入 UUID 用作代理通信信息，这个过程虽然是修改了 TLS 数据包内容（应该是同时修改了协议首部的窗口大小，即数据长度，以及协议体的数据内容），但网络拦截审查并不能知道 TLS 数据包是否被篡改，因为其没有私钥进行解密得到明文数据内容并进行 hash 运算以能够和协议头的检验和进行比对，故而没法判断数据包是否为代理流量。另外数据包到代理服务端后又去除掉了 UUID（并还原协议头），故而对于目标服务器来说 TLS 数据包并没有被篡改。

¶安装说明

服务端 trojan-go + 客户端 trojan-go

¶服务端

Trojan 实现原理：

  • 如果 TLS 握手成功，检测到 TLS 的内容非 Trojan 协议（有可能是 HTTP 请求，或者来自 GFW 的主动探测）。Trojan 将 TLS 连接代理到remote_addr:remote_port 上的 HTTP 服务。这时在远端看来，Trojan 服务就是一个 HTTPS 网站。
  • 如果 TLS 握手成功，并且被确认是 Trojan 协议头部，并且其中的密码正确，那么服务器将解析来自客户端的请求并进行代理，否则和上一步的处理方法相同。

配置说明：

  • run_type：运行类型，“client” 表示客户端，“server” 表示服务端。
  • local_addr 和 local_port：本地监听地址和端口。
  • remote_addr 和 remote_port：伪装网站的域名或 IP、伪装网站的端口。
  • password：密码，用于加密传输数据。
  • log_level：日志等级，可选项为 “trace”、“debug”、“info”、“warn”、“error"和"none”。
  • ssl：使用 SSL/TLS 加密。
  • cert 和 key：对应服务器的证书和私钥文件，用于 SSL/TLS 加密。
  • sni：SSL 证书对应的域名

配置补充说明：

  • 当 trojan 作为服务器时，local_addr 和 local_port 表示服务器监听地址和端口；remote_addr 和 remote_port 表示识别到非 trojan 协议流量时反向代理到的 HTTP 服务地址和端口（注意是 HTTP 服务，不支持 HTTPS 服务，故而实际代理的 URL 地址为：http://remote_addr:remote_port，Trojan-Go 将会测试这个 HTTP 服务是否工作正常，如果不正常，Trojan-Go 会拒绝启动）。建议反向代理到一些访问量较大的 HTTP 网站以作为伪装网站，如个人网盘等，让代理更像访问一个正常的网站，否则如果反向代理到比较简单的网站但代理流量却较大则可能会引起 GFW 怀疑。
  • 当 trojan 作为客户端时，local_addr 和 local_port 表示本地开放的 socks5/http 代理地址和端口；remote_addr 和 remote_port 表示连接服务器的地址和端口号。

¶混淆伪装（Trojan + WebSocket + TLS）

Trojan 协议本身就和 HTTPS 协议的特征和行为相同，如果没有使用 CDN 中转代理流量的需求，通常是没有必要对 Trojan 协议进行混淆的。

说明：对于主动检测，当防火墙主动连接 Trojan 服务器进行检测时，Trojan 可以正确识别非 Trojan 协议的流量。与 Shadowsocks 等代理不同的是，此时 Trojan 不会断开连接，而是将这个连接代理到一个正常的 Web 服务器。在 GFW 看来，该服务器的行为和一个普通的 HTTPS 网站行为完全相同，无法判断是否是一个 Trojan 代理节点。

trojan链接的格式为：trojan://password@remote_host:remote_port?security=tls&sni=ssl_domain&type=tcp&headerType=none#remarks。

链接参数如下：

remote_port：服务器端口号，表示Trojan服务器的监听端口号，默认为443。

sni：域名标识符，指定服务器启用SNI流量伪装的域名，一般与服务器证书中的域名相对应。

type：用于设置节点类型，可以是"ss"、“vmess”、“trojan”、"http"或"snell"等。其中，"ss"表示Shadowsocks协议，"vmess"表示V2Ray协议，"trojan"表示Trojan协议，"http"表示HTTP代理协议，而"snell"则是一种基于TLS的安全代理协议。

security：用于设置代理协议的加密方式，可选值包括"none"、“tls”、"xtls"等。其中，"none"表示不加密流量传输，"tls"表示使用基于TLS的加密方式进行传输，而"xtls"则是一种更为安全可靠的TLS扩展加密方式。

headerType：用于设置代理协议的Header类型，可以是"http"、“tls"或"none”。其中，"http"表示使用HTTP Header进行伪装，"tls"表示使用TLS Header进行伪装，而"none"则表示不进行任何Header伪装。

remarks：节点别名，无特殊作用，仅仅方便人记。

¶代理相关拓展说明

¶为什么代理软件启动后有的软件能科学上网，有的不行？

网络应用程序通常会使用默认的代理设置来发送 HTTP 请求，默认的代理设置通常为“不使用代理”（no proxy），即直接通过本地网络连接到目标服务器，不经过任何中间代理。这些应用程序不会使用系统代理的设置进行网络请求，但一般这些程序会支持手动代理设置功能，只要设置了就会根据用户自定义的代理进行网络访问。

但也会有一些应用程序（如浏览器）默认会使用系统代理的设置来发送请求。只要代理软件启动就会创建一个本地的代理服务器，并把自己的 ip 和端口号设置到系统代理设置中。如此一来，应用程序就会根据系统代理的设置将其网络请求转发给代理软件，最终代理软件再转发给外部网络的真实代理服务器，进而完成网络代理处理。

¶HTTP 代理和 SOCKS 代理的区别

HTTP 代理：HTTP 代理是最常见的代理类型，只能代理 HTTP 协议和 HTTPS 协议的请求。

SOCKS 代理：SOCKS 代理协议支持代理与任何应用层协议（例如 HTTP、SSH、FTP 等）相关的 TCP 和 UDP 流量。socks 协议是一种通用的代理协议，通过它可以将 TCP 或 UDP 连接路由到远程主机，并在本地主机上实现对连接的控制和修改。

¶具体来说，socks 代理服务器支持代理以下类型的网络流量：

  1. HTTP：通过 socks 代理服务器可以路由 HTTP 请求和响应。
  2. HTTPS：通过 socks 代理服务器可以路由 HTTPS 请求和响应。
  3. SSH：通过 socks 代理服务器可以路由 SSH 会话流量。
  4. FTP：通过 socks 代理服务器可以路由 FTP 命令和数据传输流量。
  5. DNS：通过 socks 代理服务器可以路由 DNS 查询和响应。

需要注意的是，socks 代理并不像 HTTP 代理那样支持 HTTP 缓存和其他 HTTP 优化功能，因为 socks 代理只是简单地转发网络流量，而不会对其进行解析和处理。此外，某些特定的应用层协议可能会对 socks 代理产生特殊的要求和限制，因此请根据实际情况选择最适合的代理配置方式。

¶代理时使用 socks 协议还是 http 协议？

如果需要代理所有的网络流量（包括 HTTP、HTTPS、FTP、SMTP 等），则应该使用 socks 协议。但如果只需要代理 Web 浏览器的 HTTP 流量，则使用 http 协议即可。

¶代理 HTTP 请求代理变量可以使用 Socks5 协议地址吗

比如：export http_proxy=socks5://127.0.0.1:10808

代理软件在转发数据时使用的代理协议并不是由代理变量的协议类型决定的，而是由代理软件节点配置中的协议类型决定的。比如：curl --proxy socks5://127.0.0.1:10808 https://www.google.com 这条命令，curl 会将 HTTP 协议请求数据转换为 SOCKS5 协议请求数据，如果本地没有使用代理软件，那么就会连接失败，如果有使用代理软件，则 curl 会发送给监听 10808 端口的代理软件，代理软件再根据节点配置的协议类型将 SOCKS5 协议请求数据转换为相应的协议数据最后发送给代理服务器（不是所有的代理软件都支持协议转换功能，例如 Shadowsocks 默认只支持 socks5 协议代理，如需要使用 http 协议代理则需要借助其他工具（如：privoxy）来转化本地的 http 协议数据为 socks5 协议，ShadowsocksR C# 版本则支持通过内置的混淆插件实现协议转换，而 Clash、V2ray 等则本身支持协议转换）。

¶代理软件支持哪些网络协议

一般代理软件支持应用层和传输层的各种网络协议，比如应用层的 HTTP、FTP、SMTP 等，传输层则包括 TCP 和 UDP 等。一般不支持网络层及其以下的网络通信协议层，比如网络层的 ICMP、ARP 等协议都是不支持的。要实现网络层和数据链路层的网络协议代理则需要使用 TUN 和 LAP 等技术的支持，TUN 和 LAP 技术其实就是建立本地虚拟网卡的一种技术，TUN 网卡主要处理网络层协议，TAP 网卡主要处理数据链路层协议，可以用 TUN、LAP 与代理软件配合来实现网络层和数据链路层的代理。

网络 5 层模型：

  1. 物理层：负责传输比特流，并进行电气、机械和功能性接口的定义。常见的协议有 RJ45、CLOCK、IEEE802.3 等
  2. 数据链路层：提供可靠的数据传输，通过帧同步、流量控制、差错检测等技术保证数据的正确性。常见的协议有 PPP、FR、HDLC、VLAN、MAC 等。
  3. 网络层：管理不同网络之间的数据传输，实现路由选择、拥塞控制等功能。常见的协议有 ICMP、ARP。
  4. 传输层：提供端到端的数据传输服务，包括错误校验、流量控制、拥塞避免等功能，常见的协议有 TCP 和 UDP。
  5. 应用层：提供应用程序与网络之间的接口，处理各种应用层协议，如 HTTP、FTP、SMTP 等。

¶启用 BBR 拥塞控制算法

BBR（Bottleneck Bandwidth and Round-Trip Time）是一种针对 TCP 数据流量的拥塞控制算法，可以使网络链路的带宽得到充分利用，并减少拥塞的发生。它通过动态地调整发送方的数据传输速率，以克服网络链路中的拥塞，并优化带宽利用和延迟。BBR 算法在 Linux 4.9+ 内核版本已经内置（如果内核低于 4.9 则需要升级）。用户可以根据需求进行启用。

为了方便升级内核和配置 BBR 算法，可使用秋水逸冰大佬的 BBR 一键脚本进行安装配置（支持 CentOS 6+，Debian 8+，Ubuntu 16+），参考：https://teddysun.com/489.html

说明：开启 BBR 后可能导致之前搭建的网络服务不可用，原因一般为防火墙原因。开启 BBR 后，可能会导致 Firewalld 自动停止。

¶AEAD 加密算法

AEAD 是一种同时具备保密性，完整性和可认证性的加密算法，其同时实现了加密和认证，安全性相比其他普通算法更高。

常见的 AEAD 算法如下：

  • aes-128-gcm
  • aes-192-gcm
  • aes-256-gcm
  • chacha20-ietf-poly1305
  • xchacha20-ietf-poly1305

在具备 AES 加速的 CPU（桌面，服务器）上，建议使用 aes-xxx-gcm 系列，移动设备建议使用 chacha20-ietf-poly1305 系列。

¶关于混淆代理协议的相关说明

由于代理的传输协议格式通常是固定的，GFW 可能会通过深度包检测（DPI）等技术分析流量的协议特征来判断流量是否属于代理，进而对 IP 或者端口进行封锁或限制。为了防止代理协议被识别，可以将代理流量混淆为其他常见传输协议的流量从而避免被探测识别。

代理混淆的常见传输协议有：

  • TCP（混淆原本代理流量为其他格式的 TCP 流量）
  • HTTP
  • WebSocket
  • HTTP2（也称 H2）
  • gRPC
  • QUIC
  • mKCP

为了数据包的安全，通常还会对以上协议的数据包内容进行加密（如：AEAD 加密、TLS 加密等），几种常用的 “传输协议 + 加密” 组合如下：

  • TCP + AEAD
  • HTTP + AEAD
  • WebSocket + AEAD
  • TCP + TLS
  • HTTP + TLS（HTTPS）
  • WebSocket + TLS（WSS）

TLS 流量在互联网中使用较为广泛，GFW 不可能像对 VPN 流量那样对 TLS 流量进行大面积封锁，故相对来说 TLS 代理流量不容易被 GFW 探测识别。因此，目前使用 TLS 作为传输协议的加密方式较为主流。需要说明的是：“传输协议 + TLS” 组合加密的数据都为 TLS 数据包，各种传输协议的 TLS 数据包在网络传输中的表现特征都是相似的，对于 GFW 来说，只能查看出 TLS 数据包中承载的混淆协议（如：TCP、HTTP 等），但并不能解密得到数据内容，故而无法直接知道是否为代理协议，只能通过数据包行为特征（如： TLS in TLS 特征、TLS 指纹特征等）来猜测是否为代理流量，但目前这些特征也都有相应的解决方案，故而 “传输协议 + TLS” 将会是现在和外来的绝佳代理传输组合。用户使用哪种 “传输协议 + TLS” 组合，应该取决于应用场景、性能考量等不同因素来确定，如对 CDN 中转有需求的则应该使用 HTTP + TLS 或者 WebSocket + TLS（由于 WebSocket 兼容 HTTP，故而 HTTP + TLS 也可以被 WebSocket + TLS 取代），如无 CDN 中转需求但对性能有较高需求的则因该使用 TCP + TLS。

¶TLS 四次握手

通过 TCP 三次握手建立 TCP 连接后，就可以通过 TLS 四次握手建立 TLS 连接进行数据加密传输了。

注意这里说的 TLS 四次握手并不是四次数据包发送过程，而是四次单方向过程，每次单向过程可以包括多个消息（注意：每次单向过程可以包括多个数据包，通常每个数据包可以包含多个消息），如下：

TLS 四次握手过程：

第一次握手：

  • 客户端向服务器发送一个 ClientHello 消息，其中包含支持的 TLS 版本、加密套件列表和客户端随机数（Client Random）等信息。

第二次握手：

  • 服务端从客户端提供的加密套件列表中选择一个加密套件，并向客户端发送一个 ServerHello 消息（明文传输），携带服务器支持的 TLS 版本、选择的加密套件和服务端随机数（Server Random）等信息。
  • 服务端发送服务端的证书（包含公钥）给客户端。

第三次握手：

  • 客户端验证服务端证书的有效性。
  • 如果服务端要求进行客户端证书验， 客户端会发送客户端证书给服务器进行验证（即所谓的双向认证）。
  • 客户端通过自己的私钥（如有）和服务端公钥生成一个随机数作为预秘钥（PreMaster Secret）。然后使用客户端随机数、服务端随机数以及预秘钥三个随机数通过协商的加密方式计算生成会话密钥（Session Secrect，或称为主密钥，Master Secret），同时使用服务端的公钥加密预秘钥后送给服务器。

第四次握手：

  • 服务端验证客户端证书的有效性（如需要）。
  • 服务端使用其私钥解密得到预密钥，然后使用客户端随机数、服务端随机数以及预秘钥三个随机数通过协商的加密方式计算生成会话密钥（与客户端的会话密钥相同）。
  • 服务端还会使用会话秘钥加密一段 Finish 消息发送给客户端，以验证 TLS 连接建立。

经过以上四次握手后，TLS 连接建立完成。后续数据传输中，将使用握手过程协商确定的加密套件（加密算法）通过会话密钥对传输数据进行对称加密传输。

¶TLS 代理面临的新型问题

¶TIT 特征问题

TIT 即 TLS in TLS，即进行两层 TLS 加密，进行两层 TLS 加密就会导致数据包变大（数据包变大并不是审查特征，只是降低传输效率与性能），产生两次 TLS 握手过程（属于审查特征，主要解决这个过程的问题）。常见的 VMESS 或 VLESS 或 Trojan 等 + TCP 或 WebSocket 等 + TLS 代理混淆组合通常都存在 TLS in TLS 特征。

解决方案：MITM、XTLS-Vision、 NaiveProxy

  • MITM 的原理是：仅进行 TLS 第一层加密，不进行第二层加密。使用代理服务器的 SSL 证书进行 TLS 加密，故而存在数据泄露问题（代理服务器不可信的情况）。
  • XTLS-Vision 和 NaiveProxy 的原理是：仅进行 TLS 第一层加密，不进行第二层加密，并对 TLS 握手的前 5 个特征包进行填充解决时序特征。使用目标服务器的 SSL 证书进行 TLS 加密，故而不存在数据泄露问题。

¶TLS 指纹特征

不同的软件或应用程序在 TLS 握手过程中可能会使用不同的 TLS 参数，从而导致生成不同的 TLS 指纹。因此，通过观察和分析 TLS 指纹，可以推断出正在使用的软件或应用程序（对于同个服务端软件以及同个客户端软件，产生的客户端 TLS 指纹以及服务端 TLS 指纹一般是相同的，只要维护一个 TLS 指纹数据库，流量经过时，就能对 TLS 握手数据包指纹进行分析从而判断出流量所属的服务端和客户端软件）。

参考：https://www.youtube.com/watch?v=F2VsjS6fybs&t=55s

解决办法：

  • 使用 Chromium 等常见软件的网络堆栈来伪装 TLS 指纹（实现技术如 NaiveProxy、uTls 库等），从而消除客户端 TLS 指纹特征。
  • 使用 REALITY 取代 TLS 或者使用常见 Web 服务软件（Nginx、Caddy 等）做代理的前置来反向代理，从而消除服务端 TLS 指纹特征。

说明：一些 GUI 代理客户端工具对自定义客户端 TLS 指纹支持尚不完善（如 Clash 等尚不支持、V2rayN 支持）。
